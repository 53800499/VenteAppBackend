Architecture Offline-First3.1 Principe Fondamental🔌 Offline-First — non négociableToutes les opérations métier — ventes, mouvements de stock, gestion clients, dettes — sont écrites d'abord et uniquement en local (SQLite) en V1. L'application ne dépend jamais d'une connexion réseau pour fonctionner. La synchronisation cloud (V2/V3) est strictement asynchrone : elle ne bloque jamais une action utilisateur, ne retarde aucun écran, et échoue silencieusement en arrière-plan si le réseau est indisponible — sans jamais interrompre le travail du commerçant.3.2 Les trois couches du modèle Offline-FirstCoucheRôleComportement réseau1. Écriture localeToute action (vente, stock, dette) écrit immédiatement dans SQLite via une transaction DriftAucune dépendance réseau — toujours synchrone et instantané2. File d'attente (sync_queue)Chaque écriture locale ajoute une entrée à synchroniser plus tardAucune dépendance réseau — écriture locale uniquement3. Synchronisation cloud (V2/V3)Un service en arrière-plan traite la file dès que le réseau est disponibleAsynchrone, non-bloquant, retry automatique avec backoff3.3 Règles d'implémentation Offline-FirstL'UI ne doit jamais afficher de spinner de chargement en attendant une réponse réseau pour une opération métier de base (vente, stock, client, dette).Toute opération métier est validée et confirmée à l'utilisateur dès l'écriture locale réussie — jamais après confirmation serveur.Le statut de synchronisation (sync_status) est informatif uniquement. Il ne doit jamais bloquer ou retarder une action en cours.En cas d'échec réseau prolongé, l'utilisateur peut continuer indéfiniment en local. Aucune limite de durée hors-ligne n'est imposée.Le service de synchronisation s'exécute dans un processus d'arrière-plan (background isolate Dart), jamais sur le thread UI principal.3.4 Schéma de flux Offline-FirstFlux d'une vente en mode Offline-First1. Utilisateur valide une vente2. → Transaction Drift locale (INSERT sales + sale_items + stock_movements)3. → Confirmation immédiate à l'écran (reçu affiché)4. → INSERT dans sync_queue (table_name='sales', operation='insert', payload=JSON)5. [Asynchrone, indépendant] Si réseau disponible :     → Service de sync lit sync_queue (FIFO)     → Envoi vers Supabase (V2)     → Si succès : UPDATE sync_queue SET status='done' + UPDATE sales.synced_at     → Si échec : retry_count++ (max 3, puis status='failed' + alerte discrète)6. [Si pas de réseau] : sync_queue reste 'pending' indéfiniment, sans impact utilisateur🏬  Table 1 : shopsEntité boutique — fondation du multi-boutiques (préparée dès V1)4.1 DescriptionCette table introduit l'entité Shop comme racine de toute la donnée métier.
